# v2.0 — vision

This document is **directional**, not a plan. It describes where the
standard could go once the v1.x line has matured. Nothing here is a
commitment; everything here is subject to deletion before v2 ships.

The bar for v2 is deliberately high: a breaking change is expensive,
and the only justification is a design mistake in v1 that cannot be
fixed additively.

---

## What v2 is for

A major bump happens only when the v1 data model prevents a problem
from being solved cleanly. Today, none of the v1.1 RFCs require a
major bump — they are all additive. The candidates below are the
kinds of problems that would force the conversation.

## Candidate themes

### 2.1 Monetization hooks

Today a universal-spawn manifest is a declaration of **what** ships
and **where**. It says nothing about **commerce**. A v2 manifest
could declare:

- Price points per spawn, per month, per invocation.
- A checkout flow to attach to the spawn button.
- Revenue sharing with upstream dependencies (transitive via
  `metadata.id`).
- Refund semantics and trial windows.

All of this is additive in principle. The reason it might force a v2
is that commerce fields leak into safety: a spawn that costs money is
a spawn that **must not** happen by accident, and the current
`safe_for_auto_spawn` boolean is too blunt when money is involved.

### 2.2 Cross-platform identity linking

A creation identified by `metadata.id` today has no way to prove who
owns it. In v2 an author could:

- Publish a DID (decentralized identifier) that signs the manifest.
- Link the DID to proven accounts on Claude, OpenAI, Vercel, etc.
- Receive reputation signals aggregated across platforms.

The v1.2 signing work is a prerequisite. The v2 change is the
identity model, not the cryptography.

### 2.3 Registry federation

Today the spec does not define a registry. In v2 a federation
protocol could let independent registries mirror each other:

- Each registry publishes a feed of manifests it has validated.
- Other registries consume the feed and re-index.
- A single global URL resolves a `metadata.id` to a list of canonical
  sources, ranked by the consumer's trust policy.

This does not require a central authority — it requires a protocol.

### 2.4 Capability advertisements

Today platforms differ in what they enforce: some honor
`cost_limit_usd_daily`, some only advise. A v2 mechanism would let
each platform publish a machine-readable **capability profile**:

```yaml
platform: claude
enforces:
  - min_permissions
  - rate_limit_qps
advises:
  - cost_limit_usd_daily
  - data_residency
```

Consumers could require a minimum enforcement profile before spawning.
A manifest could declare which enforcement level it needs and refuse
to spawn on platforms that don't meet it.

### 2.5 Richer dependency model

Today `dependencies[]` (in the v1.0.0 draft) references other
universal-spawn manifests by `id` and version. It is a shallow
reference. v2 could add:

- Declarative dependency graphs with version ranges resolved at
  spawn time.
- Transitive permission unions computed and shown to the user.
- Cycle detection.
- Optional pins to specific content hashes.

This overlaps with the package-manager world, but at a different
layer: v2 dependencies would describe **spawn prerequisites**, not
build-time package resolution.

### 2.6 Time-bounded manifests

A manifest could carry a validity window:

```yaml
valid_from: "2026-04-01T00:00:00Z"
valid_to:   "2027-04-01T00:00:00Z"
```

Useful for short-lived experiments, limited campaigns, or
compliance-gated creations. Today this is encoded in README prose;
promoting it to first-class would let consumers refuse to spawn stale
manifests automatically.

### 2.7 Multi-manifest repositories

A monorepo today can host one manifest at the root. v2 could allow a
lockfile-style `universal-spawn.lock` at the root that names multiple
manifests at different paths:

```yaml
members:
  - path: apps/web
  - path: apps/api
  - path: packages/sdk
```

Breaks the "one manifest at the repo root" invariant but rewards
monorepo-heavy ecosystems.

## What v2 is NOT for

- **Not** for fixing things that can be fixed in v1.x. Anything
  additive belongs in the minor track.
- **Not** for chasing trends. A change must earn its breaking status.
- **Not** for re-skinning. Cosmetic field renames alone never
  justify a major.

## When v2 ships

When, and only when, two or more candidates above reach "we cannot
express this without breaking v1." That threshold is high on purpose.

## How to propose a v2 candidate

1. Open a Discussions thread tagged `spec/v2`.
2. Argue the case for why it cannot land additively. Most proposals
   fail this test and move to v1.x.
3. If the case holds, the Editor tags `v2-candidate`. Candidates sit
   for at least six months before being accepted into a v2 draft.
4. A v2 draft lands in `spec/v2.0/` alongside the v1.x tree; old
   versions are never edited after release except for typo fixes.

## Relation to ecosystem churn

The upstream ecosystem — AI model generations, new hosting providers,
new game engines — churns faster than this spec. We do **not** bump
majors just because Claude ships a new model generation or Vercel
ships a new runtime. Those are platform-extension updates, fully
additive.
