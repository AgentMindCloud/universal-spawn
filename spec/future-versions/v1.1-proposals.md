# v1.1 proposals in flight

Concrete RFCs targeting the next minor release. Each item here either
lands in v1.1 or moves to `v2.0-vision.md`. All proposals **MUST** be
additive; breaking changes wait for v2.

Track status in the issue tracker under the
[`spec/v1.1`](https://github.com/AgentMindCloud/universal-spawn/labels/spec%2Fv1.1)
label.

---

## RFC-001 — Richer platform extension fields

**Status**: draft.
**Goal**: expand the thin-est extension schemas to cover common needs
authors hit in the first month.

Concrete additions under consideration:

- `platforms.claude` — a `skills[]` array to bundle multiple skills in
  one manifest; a `mcp_server` block with `transport`, `command`,
  `args`, `env` for MCP registration.
- `platforms.openai` — `assistants[]` and `agents[]` blocks for the
  Agents / Assistants APIs.
- `platforms.vercel` — `headers[]`, `redirects[]`, `rewrites[]`,
  `crons[]` that mirror `vercel.json` so the Vercel manifest can be
  deprecated over time.
- `platforms.netlify` — a parallel `redirects[]`, `headers[]` block.
- `platforms.huggingface` — `inference.endpoints[]` expansion for
  Inference Endpoints deployment.

No core-schema changes required — these all land under platform
extension schemas. The list of registered platforms is the only
cross-cutting change and is already allowed by the core schema.

## RFC-002 — Per-platform validator SDK

**Status**: scoping.

A single Node / Python package that accepts a manifest plus a platform
id and produces:

- Core schema validation (draft-07).
- Platform extension validation.
- Platform-specific lint (e.g. Claude tool names match the function
  refs; Vercel `functions[].match` globs resolve to actual files).

Ships as `@universal-spawn/validator` on npm and
`universal-spawn-validator` on PyPI. Conformance test suite shared
between both implementations.

## RFC-003 — Reference registry

**Status**: scoping.

A read-only index that crawls public GitHub repositories containing a
`universal-spawn.yaml`. Provides:

- Search by `name`, `type`, `keywords`, `categories`, `metadata.id`.
- Per-manifest validation status badge.
- Permalink back to the source repository at a specific commit.

**Not** a package manager. No artifacts hosted. No downloads proxied.

## RFC-004 — Canonical platform list in the core spec

**Status**: draft.

Today the core spec says "platforms are registered separately." RFC-004
adds a lightweight registration table directly in the spec, listing
the id, the steward, and the extension schema URL. The table is
purely informational; it does not constrain the core schema.

## RFC-005 — IANA media type registration

**Status**: scoping.

Submit registration for:

- `application/vnd.universal-spawn+yaml`
- `application/vnd.universal-spawn+json`
- `application/vnd.universal-spawn+toml`

Prepares the ground for HTTP discovery and content-negotiated serving
of manifests by registries.

## RFC-006 — Declarative visual asset requirements

**Status**: draft.

Add `visuals.icon_min_dimensions`, `visuals.hero_plate_aspect_ratio`,
and a `visuals.color_contrast_min` hint so downstream consumers can
decide whether a manifest's visuals meet a surface's minimum bar (a
card that shows a 40×40 icon can reject anything smaller than that
claimed dimension).

All fields optional. All additive.

## RFC-008 — `deployment.artifacts[]`

**Status**: draft.

Some creations produce artifacts that ship to multiple registries
(npm + Homebrew + Cargo is a common CLI case). RFC-008 adds
`deployment.artifacts[]`:

```yaml
deployment:
  artifacts:
    - registry: npm
      name: "@universal-spawn/lint"
    - registry: homebrew
      formula: "universal-spawn-lint"
    - registry: cargo
      crate: "universal-spawn-lint"
```

Additive. Strict enum of registries. Default empty.

## RFC-009 — `compat` surface for other standards

**Status**: draft.

Parallel to `x-ext.org.agentmindcloud.grok-install`, a first-class
`compat` object with named sub-keys:

```yaml
compat:
  grok_install:
    version: "2.14"
  openapi:
    path: openapi.yaml
  dockerfile:
    path: Dockerfile
```

This lifts a handful of common cross-standard references into the
core spec. Additive.

## How an RFC lands

1. Opens as a Discussions thread with the RFC template.
2. Receives public comment for at least 14 days.
3. Two maintainer approvals from different platform stewards.
4. Editor tags `accepted`.
5. Lands in `spec/v1.1/` under the RFC number.
6. `spec/v1.1/CHANGELOG.md` entry cites the RFC id.

Proposals that cannot achieve consensus move here to
`v1.1-proposals.md → not-accepted` or to `v2.0-vision.md` for
consideration in a later major.
